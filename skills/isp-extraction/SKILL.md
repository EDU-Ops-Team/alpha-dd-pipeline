# ISP Extraction (AGENT-08)

## Purpose

Extract structured data from an ISP (Interior Space Plan) document — specifically the "Program Fit Analysis" PDF generated by the Alpha space-planning tool. The output feeds downstream work units: RayCon cost estimates (WU-10), Opening Plan (WU-12), and DD Report Assembly (WU-13).

This agent reads a completed document. It does not run the space-planning tool, generate floor plans, or make capacity judgments. It extracts what the document says.

## Input

- ISP PDF file (from Drive upload or email attachment)
- `site_meta` from Sindri (for address matching and site_id)

## Document Format

The ISP is a "Program Fit Analysis" PDF, typically 10–15 pages. It has a fixed section structure:

| Section | Page(s) | What It Contains |
|---|---|---|
| Building Code Information | 1 | Code edition, occupancy class, jurisdiction, amendments, sprinkler status, total occupant load, gross floor area |
| Executive Summary | 1 | Overall score (0–100), fit rating (GOOD FIT / MARGINAL FIT / POOR FIT), requirements met ratio, requirements score + quality score |
| Key Metrics | 1 | Total rooms, rooms assigned/unassigned, target capacity, recommended capacity, avg fit score, best tier met |
| Capacity Analysis | 2 | Grade span, guides required, recommended capacity, gross ceiling, NLA capacity, effective SF/student, sharing penalty, space requirements met (workshop, 1:1 meeting, play area, dining/commons) |
| Classroom Assignments | 2 | Table: room ID → grade level → student count → area SF. Plus per-level totals |
| Tier Evaluation | 3 | Table: tier name → meets? → required met ratio → assignment % → fit % → missing items |
| Floor Plan Visualizations | 4–5 | Color-coded floor plan images per floor (visual only — no extractable data beyond room labels) |
| ADA Pre-Check | 6–7 | ADA score (0–100), error count, warning count. Violations by Rule summary table. All Violations detail table (severity, location, rule, actual value, required value, description) |
| IBC Compliance | 8–9 | IBC score (0–100), error/warning counts, total occupant load. Occupant Load by Room table (room ID, type, area, factor, method, load). Plumbing Fixture Requirements table (fixture type, required count, notes). Plumbing summary line. IBC Violations tables |
| Adjacency Compliance | 10 | Adjacency score (0–100), critical/high/medium counts. Adjacency violations table (priority, rule #, room types, relationship, detail) |
| Requirement Status | 11 | Room type → required count → assigned count → status (Met/Not Met) |
| Optimization Proposals | 11–12 | Numbered swap/move proposals with room IDs, resulting SF, reason, priority, fit delta |
| Room Schedule | 13 | Complete room inventory: room ID, floor, assignment, area SF, dimensions, occupant load, fit score |
| Door Schedule | 14 | Complete door inventory: door ID, floor, width (in), height (in), notes (violations) |

## Extraction Process

### Step 1: Building Code Information

Extract from the first-page table:

| Field | Extract As |
|---|---|
| Applicable Code | `building_code` — string (e.g., "IBC 2018") |
| Occupancy Classification | `occupancy_classification` — string (e.g., "Group E (Educational)") |
| Jurisdiction | `jurisdiction` — string (e.g., "OK:Oklahoma") |
| Amendments Applied | `amendments` — string (full text) |
| Sprinkler System | `sprinkler_system` — boolean (Yes → true, No → false) |
| Total Occupant Load | `total_occupant_load` — integer |
| Gross Floor Area | `gross_floor_area_sf` — integer (strip "sqft" suffix) |

### Step 2: Executive Summary and Key Metrics

Extract:

| Field | Extract As |
|---|---|
| Score | `program_fit_score` — integer 0–100 |
| Rating | `program_fit_rating` — string ("GOOD FIT", "MARGINAL FIT", "POOR FIT") |
| Requirements Met | `requirements_met` — string (e.g., "9/9") |
| Requirements Score | `requirements_score` — string (e.g., "60/60") |
| Quality Score | `quality_score` — string (e.g., "38/40") |
| Total Rooms Detected | `total_rooms` — integer |
| Rooms Assigned | `rooms_assigned` — integer |
| Rooms Unassigned | `rooms_unassigned` — integer |
| Target Capacity | `target_capacity` — integer or null (if "N/A") |
| Recommended Capacity | `recommended_capacity` — integer (strip "students" suffix) |
| Avg Fit Score | `avg_fit_score` — float as percentage (e.g., 89.3) |
| Best Tier Met | `best_tier_met` — string (e.g., "ideal", "absolute_min") |

### Step 3: Capacity Analysis

Extract from the Capacity Analysis table:

| Field | Extract As |
|---|---|
| Grade Span | `grade_span` — string (e.g., "Grades K-1, 2-3, 4-5, 6-8") |
| Guides Required | `guides_required` — integer |
| Gross Ceiling (100 SF/student) | `gross_ceiling_capacity` — integer |
| NLA Capacity | `nla_capacity` — integer |
| Effective SF/Student | `effective_sf_per_student` — integer (strip "SF/student") |
| Sharing Penalty | `sharing_penalty` — string ("None" or description) |

Extract space requirement status as an object:

```
"space_requirements": {
  "workshop": "Met | Not Met",
  "one_on_one_meeting": "Met | Not Met",
  "play_area": "Met | Not Met",
  "dining_commons": "Met | Not Met"
}
```

### Step 4: Classroom Assignments

Extract each classroom row plus the per-level totals:

```
"classroom_assignments": [
  {
    "room_id": "R2",
    "level": "K-1",
    "students": 3,
    "area_sf": 579
  }
],
"level_totals": [
  {
    "level": "K-1",
    "room_count": 2,
    "total_students": 6
  }
]
```

### Step 5: Tier Evaluation

Extract each tier row:

```
"tier_evaluation": [
  {
    "tier": "ideal",
    "meets": true,
    "required_met": "9/9",
    "assignment_pct": 100.0,
    "fit_pct": 89.3,
    "missing": null
  }
]
```

### Step 6: ADA Pre-Check

Extract the ADA summary and all violations:

```
"ada_precheck": {
  "score": 83,
  "errors": 6,
  "warnings": 7,
  "violations_by_rule": [
    {
      "rule": "Door Clear Width",
      "count": 6,
      "severity_breakdown": "6 error(s)"
    }
  ],
  "violations": [
    {
      "severity": "ERROR",
      "location": "D5",
      "rule": "Door Clear Width",
      "actual": "21\"",
      "required": "32\"",
      "description": "Door D5 clear width is 21.0\" but minimum is 32\""
    }
  ]
}
```

**Location normalization:** The document uses internal IDs for door locations (e.g., "7ys138aesaybbc45yitqfw5cd"). When a human-readable door ID can be inferred from the description (e.g., "Door D5"), use the door ID. When only the internal ID is available and no door ID appears in the description, preserve the internal ID.

### Step 7: IBC Compliance

Extract the IBC summary, occupant load table, plumbing requirements, and violations:

```
"ibc_compliance": {
  "score": 99,
  "errors": 0,
  "warnings": 1,
  "total_occupant_load": 181,
  "occupant_load_by_room": [
    {
      "room_id": "R1-S1",
      "type": "DINING",
      "area_sf": 361,
      "factor": 15,
      "method": "net",
      "load": 24
    }
  ],
  "plumbing_fixtures": {
    "water_closets_male": { "required": 2, "notes": "1 per 50 male occupants (Group E)" },
    "water_closets_female": { "required": 2, "notes": "1 per 50 female occupants (Group E)" },
    "lavatories": { "required": 4, "notes": "1 per 50 occupants" },
    "drinking_fountains": { "required": 2, "notes": "1 per 100 occupants" }
  },
  "plumbing_summary": "string — the summary line (e.g., 'Building has 3 restroom(s) totaling 165 SF...')",
  "violations": [
    {
      "severity": "WARNING",
      "rule": "Plumbing Fixture Shortfall",
      "description": "string — full description"
    }
  ]
}
```

### Step 8: Adjacency Compliance

Extract adjacency findings:

```
"adjacency_compliance": {
  "score": 53,
  "critical_count": 2,
  "high_count": 4,
  "medium_count": 1,
  "violations": [
    {
      "priority": "Critical",
      "rule_number": 26,
      "room_types": "RESTROOM_CHILDREN",
      "relationship": "must be distributed",
      "detail": "RESTROOM_CHILDREN present on 1 of 2 occupied floors; missing on: Floor 2"
    }
  ]
}
```

### Step 9: Requirement Status

Extract the requirement status table:

```
"requirement_status": [
  {
    "room_type": "Classroom",
    "required": "4+ (required)",
    "assigned": 6,
    "status": "Met"
  }
]
```

### Step 10: Optimization Proposals

Extract each proposal:

```
"optimization_proposals": [
  {
    "number": 1,
    "action": "SWAP",
    "rooms": "R2 + R1-S4",
    "result_type": "FOOD_SERVING",
    "result_sf": 883,
    "reason": "Satisfy adjacency rule 13 (must be separated)",
    "priority": "high",
    "fit_delta": -0.65
  }
]
```

### Step 11: Room Schedule

Extract the complete room inventory. This is the most important table for downstream work units:

```
"room_schedule": [
  {
    "room_id": "R1-S1",
    "floor": "Floor 2",
    "assignment": "DINING",
    "area_sf": 361,
    "dimensions": "10 x 53",
    "occupant_load": 24,
    "fit_score_pct": 96
  }
]
```

**Rules for room_schedule:**
- `occupant_load`: integer or null (rooms like restrooms, corridors show "-" → null)
- `fit_score_pct`: integer (strip "%" suffix)
- This is the authoritative room list — all other room references should cross-check against it

### Step 12: Door Schedule

Extract the complete door inventory:

```
"door_schedule": [
  {
    "door_id": "D1",
    "floor": "Floor 1",
    "width_in": 35.2,
    "height_in": 83.9,
    "notes": null
  }
]
```

**Rules for door_schedule:**
- `width_in` and `height_in`: float
- `notes`: string or null (null when document shows "-")
- Doors with notes contain ADA or code violations — these map to the ADA Pre-Check violations

## Output

### Sindri Data Out: `isp_extract`

Write the full schema defined in `references/isp-extraction-schema.json` to Sindri.

### Human-Readable Output

In addition to the structured JSON, produce a human-readable markdown summary saved to the site's Drive folder. This summary must be self-contained and readable by someone who hasn't seen the PDF. Structure:

```markdown
# Program Fit Analysis — {address}
Generated: {date} | Extracted: {timestamp}

## Score Summary
- **Program Fit:** {score}/100 — {rating}
- **IBC Compliance:** {ibc_score}/100
- **ADA Pre-Check:** {ada_score}/100
- **Adjacency:** {adjacency_score}/100

## Capacity
- Recommended: {recommended_capacity} students
- Grade span: {grade_span}
- Guides required: {guides_required}
- Effective SF/student: {effective_sf_per_student}

## Room Inventory ({total_rooms} rooms)
[Room schedule table: room_id | floor | assignment | area SF | occupant load | fit score]

## Classroom Assignments
[Classroom assignments table with level totals]

## Code Issues
### ADA ({ada_errors} errors, {ada_warnings} warnings)
[List errors first, then warnings]

### IBC ({ibc_errors} errors, {ibc_warnings} warnings)
[List violations]

### Adjacency ({critical} critical, {high} high, {medium} medium)
[List violations by priority]

## Plumbing Requirements
[Fixture requirements table + summary of shortfall if any]

## Optimization Proposals
[Numbered list of proposals]
```

Save to: `reports/{address-kebab-case}_isp-extract_{YYYY-MM-DD}.md`

## Sindri Schema Key Fields for Downstream

The DD Report Assembly (WU-13) and RayCon Cost Estimates (WU-10) primarily use these fields:

| Downstream WU | Fields Used |
|---|---|
| WU-10 (RayCon) | `room_schedule` (room inventory for scope estimation), `ibc_compliance.plumbing_fixtures` (fixture gaps), `ada_precheck.violations` (ADA scope items) |
| WU-12 (Opening Plan) | `recommended_capacity`, `program_fit_score`, `program_fit_rating`, `ibc_compliance.violations` (code triggers) |
| WU-13 (DD Report) | `recommended_capacity`, `program_fit_score`, `room_schedule`, `classroom_assignments`, `level_totals`, `ibc_compliance.plumbing_fixtures`, `ada_precheck` summary, `adjacency_compliance` summary |

## Hard Rules

1. **Never fabricate room data.** Extract only what the document states. If a field is missing from the document, set it to null.
2. **Never recalculate occupant loads.** The document contains computed occupant loads per room — extract the document's values, do not re-derive from area and factor.
3. **Never recalculate scores.** Extract the document's stated scores (program fit, IBC, ADA, adjacency). Do not attempt to verify or recalculate them.
4. **Preserve the document's room IDs exactly.** Room IDs like "R1-S3", "T1-S1", "R10" are identifiers used across sections. Do not rename or normalize them.
5. **The Room Schedule is the source of truth** for room inventory. If a room appears in the Room Schedule but not in Classroom Assignments, it still exists — it's just not assigned as a classroom. If a room appears in Classroom Assignments but not in Room Schedule, flag the discrepancy.
6. **Door violations map to ADA violations.** Cross-reference the Door Schedule notes with the ADA Pre-Check violations. Both should show the same issues — if they don't, flag the discrepancy.
7. **Occupant load null for non-occupied rooms.** Restrooms, corridors, janitorial, and IT rooms typically show "-" for occupant load. Extract as null, not zero.
8. **Fit score is per-room, not per-student.** The fit_score_pct applies to the room's suitability for its assigned function, not a per-student measure.
9. **Floor plan images are not extractable data.** Pages 4–5 (floor plan visualizations) contain spatial layout information that cannot be reliably extracted as text. Skip these pages. The Room Schedule on page 13 contains the same room data in structured form.
10. **Tier names are document-specific.** Extract tier names exactly as they appear (e.g., "ideal", "absolute_min"). Do not normalize or rename them.
11. **All scores are integers.** Program fit, IBC, ADA, and adjacency scores are all 0–100 integers.
12. **Fit delta in optimization proposals can be negative.** A negative fit_delta means the swap reduces overall fit score. Extract the sign as-is.

## Quality Bar

The extraction is complete when:

- All 12 extractable sections have been processed (skip floor plan images)
- Every room in the Room Schedule appears in the JSON room_schedule array
- Every door in the Door Schedule appears in the JSON door_schedule array
- Every ADA violation appears in the ada_precheck.violations array
- Every IBC violation appears in the ibc_compliance.violations array
- Every adjacency violation appears in the adjacency_compliance.violations array
- The requirement_status array covers all room types listed in the document
- classroom_assignments entries cross-check against room_schedule entries
- A human-readable markdown summary has been generated
- The structured JSON validates against `references/isp-extraction-schema.json`

## Not In Scope

- Running the space-planning tool
- Generating or modifying floor plans
- Recalculating capacity, occupant loads, or scores
- Making judgments about room suitability — that's the ISP tool's job
- Extracting data from floor plan images (pages 4–5)
